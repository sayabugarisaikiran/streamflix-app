📡 Route 53 Deep Dive

Understanding DNS record types, IP ↔ DNS mapping, and routing policies

Record Types: IP → DNS vs DNS → DNS

A Record Domain → IP Address
streamflix.com
→
54.230.10.42

Maps a domain name directly to an IPv4 address. Used when you know the exact IP of your server (e.g., an EC2 instance with an Elastic IP).

streamflix.com → A → 54.230.10.42
AAAA Record Domain → IPv6 Address
streamflix.com
→
2600:1f18:...:8a

Same as A record but for IPv6 addresses. Modern infrastructure increasingly uses IPv6.

streamflix.com → AAAA → 2600:1f18:2551:8a00::1
CNAME Record DNS Name → DNS Name
www.streamflix.com
→
streamflix.com

Maps one domain name to another. Cannot be used at the zone apex (root domain like streamflix.com). Use for subdomains only!

www.streamflix.com → CNAME → streamflix.com
⚠️ CNAME replaces ALL records at that name. Cannot coexist with other types.
ALIAS Record ⭐ DNS → AWS Resource (Free!)
streamflix.com
→
d3abc.cloudfront.net

AWS-specific! Works at the zone apex. Points to AWS resources (CloudFront, ALB, S3, etc.). Free of charge — no Route 53 query fees. This is the preferred way!

streamflix.com → ALIAS (A) → d3abc.cloudfront.net

Other Important Record Types

MX

Mail Exchange

Routes email to mail servers

streamflix.com → MX 10 → mail.google.com
TXT

Text Record

Domain verification, SPF, DKIM for email

streamflix.com → TXT → "v=spf1 include:_spf.google.com"
NS

Name Server

Delegates a hosted zone to specific DNS servers

streamflix.com → NS → ns-1234.awsdns-12.org
SOA

Start of Authority

Zone metadata: primary NS, admin email, serial number

Auto-created with every hosted zone
CAA

Certificate Authority Auth

Controls which CAs can issue SSL certs for your domain

streamflix.com → CAA → 0 issue "amazon.com"
SRV

Service Locator

Specifies host/port for services (SIP, LDAP, etc.)

_sip._tcp.streamflix.com → SRV → 10 5 5060 sip.example.com

Routing Policies — How Route 53 Decides Where to Send Traffic

Simple Default · No Intelligence

The most basic routing. Maps a domain name to one or more resources. If you specify multiple IPs, Route 53 returns them all — the client picks one randomly. No health checks.

Traffic Flow
User
→
Server A
📋 How It Works
  1. User queries streamflix.com
  2. Route 53 returns the IP(s) you configured
  3. If multiple IPs → client picks randomly
🏢 Real-World Example

A small company blog hosted on a single EC2 instance with an Elastic IP. No need for complex routing.

⚙️ AWS Console Config
Record name: streamflix.com | Type: A | Value: 54.230.10.42 | Routing policy: Simple
⚠️ No health checks! If the server dies, DNS still points to it.
Weighted Traffic Splitting · A/B Testing

Distributes traffic based on weights you assign. Route 53 calculates the percentage as: weight / total_weight. Setting a weight to 0 stops all traffic to that record.

Traffic Flow (70/30 Split)
User
→
v2.0 (new) 70%
v1.0 (old) 30%
📋 How It Works
  1. Create 2+ records with the same name
  2. Assign weights: v2 = 70, v1 = 30
  3. 70% of queries → v2, 30% → v1
  4. Set v1 weight = 0 to finish migration
🏢 Real-World Example

Canary deployment: Netflix deploys a new version to 5% of users. If no errors after 1 hour, increase to 50%, then 100%.

⚙️ AWS Console Config
Record 1: A | Value: 10.0.1.10 | Weight: 70 | Set ID: "v2-prod"
Record 2: A | Value: 10.0.2.20 | Weight: 30 | Set ID: "v1-prod"
💡 Supports health checks. If a weighted target fails, Route 53 removes it from rotation automatically.
Latency Fastest Region · Global Performance

Routes each user to the AWS region with the lowest network latency to them. AWS maintains a global latency database — it's not measured live per request.

User in Mumbai → Nearest Region
India 🇮🇳
→
Mumbai 12ms ✓
Virginia 210ms
Ireland 180ms
📋 How It Works
  1. Deploy app in multiple AWS regions
  2. Create latency records for each region
  3. User's DNS resolver asks Route 53
  4. Route 53 checks latency table → picks lowest
🏢 Real-World Example

Amazon.com uses this! Users in India get routed to ap-south-1 (Mumbai), US users to us-east-1. Reduces page load time by 200ms+.

⚙️ AWS Console Config
Record 1: A | Value: 13.235.x.x | Region: ap-south-1 | Set ID: "mumbai"
Record 2: A | Value: 54.230.x.x | Region: us-east-1 | Set ID: "virginia"
🔍 vs Geolocation: Latency = which server responds fastest. Geolocation = where the user physically is. A user in India might get fastest response from Singapore, not Mumbai.
Failover Disaster Recovery · Active-Passive

Creates an active-passive pair. Route 53 health-checks the primary. If the primary fails 3 times in a row, traffic automatically switches to the secondary. No manual intervention.

Primary fails → Auto-switch to standby
User
→
Primary (us-east-1) ❌
Standby (eu-west-1) ✅
📋 How It Works
  1. Create a health check for primary endpoint
  2. Primary record → Failover type = PRIMARY
  3. Secondary record → Failover type = SECONDARY
  4. Health check fails 3x → Route 53 switches to secondary
  5. Primary recovers → Route 53 switches back
🏢 Real-World Example

Banking website: Primary in us-east-1, standby in eu-west-1. If Virginia goes down (like the 2017 S3 outage), customers automatically hit Ireland. Zero downtime.

⚙️ AWS Console Config
Record 1: A | Value: ALB-Primary | Failover: Primary | Health Check: hc-xyz
Record 2: A | Value: ALB-Standby | Failover: Secondary
💡 The secondary can be a static S3 website showing "We're experiencing issues, please wait" — this is a common pattern!
Geolocation Country-Based · Content Licensing

Routes based on where the user physically is (country, continent, or US state). NOT about speed — this is about showing different content based on location.

Different countries → Different servers
🇯🇵 Japan → jp.streamflix.com
🇩🇪 Germany → eu.streamflix.com
🌎 Default → us.streamflix.com
📋 How It Works
  1. Create records with location rules:
    • Country = Japan → Tokyo ALB
    • Continent = Europe → Ireland ALB
    • Default → Virginia ALB
  2. Route 53 detects user's country from DNS resolver IP
  3. Matches the most specific rule first
🏢 Real-World Example

Netflix content licensing: "Squid Game" is available in Korea but not in the US. Geo-routing ensures Korean users see Korean content library, US users see US library.

GDPR compliance: EU users are routed to EU-based servers that comply with European data laws.

⚙️ AWS Console Config
Record 1: A | Value: 13.112.x.x | Location: Asia - Japan | Set ID: "japan"
Record 2: A | Value: 34.245.x.x | Location: Europe | Set ID: "europe"
Record 3: A | Value: 54.230.x.x | Location: Default | Set ID: "default"
⚠️ Always create a Default record! Without it, users from unlisted locations get NO response (NXDOMAIN error).
🔍 vs Latency: Geolocation = "You ARE in India → go to India server." Latency = "You GET fastest response from Singapore → go there." Geolocation is legal/content-driven, Latency is performance-driven.
Multivalue DNS Load Balancing · Health-Aware

Returns up to 8 healthy IP addresses in response to a DNS query. The client randomly picks one. Like Simple routing BUT with health checks, so dead servers are excluded.

Returns only healthy IPs
User
→
10.0.1.10 ✅
10.0.2.20 ✅
10.0.3.30 ❌
📋 How It Works
  1. Create 8 A records, each with its own health check
  2. Route 53 checks health of all 8
  3. Returns only healthy ones (up to 8)
  4. Client picks one randomly
🏢 Real-World Example

Small startup with 4 web servers and no budget for an ALB ($16/month). Use Multivalue to spread traffic and auto-remove bad servers from DNS.

⚙️ AWS Console Config
Record 1: A | Value: 10.0.1.10 | Set ID: "web-1" | Health Check: hc-1
Record 2: A | Value: 10.0.2.20 | Set ID: "web-2" | Health Check: hc-2
Record 3: A | Value: 10.0.3.30 | Set ID: "web-3" | Health Check: hc-3
🔍 vs Simple: Simple returns ALL IPs (including dead ones). Multivalue only returns healthy IPs. vs ALB: ALB is true load balancing (Layer 7, sticky sessions, path routing). Multivalue is basic DNS-level distribution.
Geoproximity Distance + Bias · Traffic Flow Required

Routes based on geographic distance between user and resource, with a configurable bias to expand or shrink a region's "catchment area." Available only in Route 53 Traffic Flow (visual editor).

Bias expands Mumbai's coverage area
Mumbai (bias: +25) ← attracts more traffic
Virginia (bias: 0) ← normal area
Ireland (bias: -10) ← shrinks area
📋 How It Works
  1. Open Route 53 → Traffic Flow → Create policy
  2. Add resources with lat/long or AWS region
  3. Adjust bias (−99 to +99)
  4. Positive bias = larger area = more traffic
  5. Negative bias = smaller area = less traffic
🏢 Real-World Example

Expanding into a new market: You launch Mumbai servers but want to gradually shift Middle East traffic from Ireland → Mumbai. Increase Mumbai bias from 0 → +25 → +50 over weeks.

⚙️ AWS Console Config
This policy is ONLY available via Route 53 Traffic Flow (visual editor).
Costs: $50/month per Traffic Flow policy.
🔍 vs Geolocation: Geolocation = hard country boundaries. Geoproximity = flexible distance-based with adjustable bias. You can "steal" traffic from nearby regions.
IP-Based CIDR Routing · ISP Optimization

Routes based on the client's IP address range (CIDR blocks). You define which IP ranges should go to which endpoints. Useful for ISP-specific or enterprise network optimization.

Different IP ranges → Different endpoints
203.0.113.0/24 → ISP-A endpoint
198.51.100.0/24 → ISP-B endpoint
Default → General endpoint
📋 How It Works
  1. Create a CIDR collection in Route 53
  2. Add CIDR blocks (e.g., 203.0.113.0/24)
  3. Associate CIDR blocks with DNS records
  4. Route 53 matches client IP → routes to matched record
🏢 Real-World Example

Telecom company: Jio users (specific IP ranges) are routed to Mumbai CDN, Airtel users to a different CDN. Optimizes peering and reduces inter-ISP transit costs.

⚙️ AWS Console Config
Step 1: Create CIDR collection → Add CIDR locations
Step 2: Create record → Routing: IP-based → Select CIDR location
💡 Newest routing policy (added 2022). Rarely asked in interviews but very useful for ISPs and large enterprises.
