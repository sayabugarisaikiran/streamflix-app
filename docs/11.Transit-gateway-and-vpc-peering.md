# SECTION A: TRANSIT GATEWAY

## Part 1: Why Transit Gateway Exists (15 minutes)

### 🗣️ Opening — The Problem

*"In the last session we learned VPC Peering. Quick recap: Peering connects two VPCs so they can talk privately. Works great."*

*"But here's what happens in the real world. A company starts with 2 VPCs — Dev and Prod. They peer them. Easy."*

*"6 months later, they need a Staging VPC. Now 3 peerings."*
*"Then a Shared Services VPC for logging and monitoring. 6 peerings."*
*"Then a Security VPC, a Data VPC, and their acquired company's VPCs..."*

```
2 VPCs  = 1 peering
3 VPCs  = 3 peerings
5 VPCs  = 10 peerings
10 VPCs = 45 peerings
20 VPCs = 190 peerings
50 VPCs = 1,225 peerings  ← Unmanageable!
```

*"Formula: n × (n-1) / 2. For 50 VPCs, that's 1,225 peering connections. Each needs its own route table entries on both sides. That's 2,450 route updates. One mistake = security breach. This is why enterprises need Transit Gateway."*

---

### 🗣️ The Airport Analogy

*"VPC Peering is like direct flights. Mumbai to Delhi? One flight. Mumbai to Bangalore? Another flight. To connect 10 cities directly, you need 45 flight routes. Expensive, hard to manage."*

*"Transit Gateway is like a HUB AIRPORT. Instead of 45 direct flights, every city flies to one hub (like Dubai). From the hub, you can reach any city. 10 cities = 10 routes. Simple."*

```
WITHOUT Transit Gateway (Mesh):      WITH Transit Gateway (Hub):

  VPC-A ─── VPC-B                        VPC-A ─┐
    │  ╲  ╱  │                           VPC-B ─┤
    │   ╲╱   │                           VPC-C ─┼── Transit Gateway
    │   ╱╲   │                           VPC-D ─┤     │
    │  ╱  ╲  │                           VPC-E ─┘     │
  VPC-C ─── VPC-D                                     │
    ╲      ╱                              On-Prem ─── VPN
     VPC-E
  (10 connections)                       (6 connections)
```

## Part 2: What is Transit Gateway? (15 minutes)

### 🗣️ Definition

*"AWS Transit Gateway (TGW) is a regional network hub that connects VPCs, VPN connections, Direct Connect gateways, and even other Transit Gateways together through a single, central point."*

### 🗣️ Key Facts

| Fact | Detail |
|------|--------|
| **Regional** | Lives in one region. But you can PEER Transit Gateways across regions. |
| **Scalable** | Supports up to 5,000 VPC attachments per TGW |
| **Bandwidth** | Up to 50 Gbps per VPC attachment (burst) |
| **Routing** | Has its OWN route tables — separate from VPC route tables |
| **Transitive** | Unlike VPC peering, TGW IS transitive. VPC-A → TGW → VPC-B works automatically. |
| **AWS Managed** | Highly available, redundant, no patching |

### 🗣️ What Can Attach to Transit Gateway?

```
                    ┌────────────┐
    VPC-Dev ───────►│            │
    VPC-Staging ───►│  Transit   │◄─── AWS Direct Connect
    VPC-Prod ──────►│  Gateway   │◄─── Site-to-Site VPN
    VPC-Shared ────►│            │◄─── Another TGW (peering)
                    └────────────┘
```

| Attachment Type | What It Is |
|-----------------|-----------|
| **VPC Attachment** | Connects a VPC to the TGW |
| **VPN Attachment** | Connects a Site-to-Site VPN tunnel |
| **Direct Connect Gateway** | Connects a dedicated physical line |
| **TGW Peering** | Connects to a TGW in another region (inter-region) |
| **Connect Attachment** | For SD-WAN appliances (GRE/BFD) |

### ❓ Ask Students:

*"What's the fundamental difference between VPC Peering and Transit Gateway?"*

*"Answer: VPC Peering is NOT transitive — if A peers with B and B peers with C, A cannot reach C. Transit Gateway IS transitive — all attachments can reach each other (unless you restrict it with route tables)."*

## Part 3: Transit Gateway Architecture Deep Dive (20 minutes)

### 🗣️ TGW Route Tables

*"Transit Gateway has its OWN routing system — completely separate from VPC route tables. There are two concepts:"*

**1. Associations** — Which attachment uses which TGW route table for OUTGOING traffic?
**2. Propagations** — Which attachments ADVERTISE their routes to which TGW route table?

```
TGW Route Table: "main-rt"
┌──────────────────────┬──────────────────┐
│ Destination          │ Target           │
├──────────────────────┼──────────────────┤
│ 10.0.0.0/16          │ VPC-Dev          │  ← Propagated from VPC-Dev
│ 10.1.0.0/16          │ VPC-Staging      │  ← Propagated from VPC-Staging
│ 10.2.0.0/16          │ VPC-Prod         │  ← Propagated from VPC-Prod
│ 192.168.0.0/16       │ VPN-OnPrem       │  ← Propagated from VPN
│ 0.0.0.0/0            │ VPC-Egress       │  ← Static route to shared internet
└──────────────────────┴──────────────────┘
```

*"When VPC-Dev sends a packet to 10.2.0.0/16 (Prod), the TGW checks its route table, sees it maps to VPC-Prod's attachment, and forwards it."*

### 🗣️ Default Behavior vs Isolated

**Default (All-talk):**
*"By default, all attachments share ONE route table. Everyone can talk to everyone."*

```
All VPCs → Associated with "default-rt"
All VPCs → Propagate to "default-rt"
Result: VPC-A ↔ VPC-B ↔ VPC-C ↔ VPN  (full mesh through TGW)
```

**Isolated (Segmented):**
*"In production, you DON'T want Dev talking to Prod. You create multiple TGW route tables:"*

```
TGW Route Table: "shared-rt"
  Associated: VPC-Shared-Services
  Propagated: VPC-Dev, VPC-Staging, VPC-Prod, VPN
  → Shared Services can reach ALL VPCs

TGW Route Table: "dev-rt"
  Associated: VPC-Dev
  Propagated: VPC-Shared-Services only
  → Dev can ONLY reach Shared Services, NOT Prod or Staging

TGW Route Table: "prod-rt"
  Associated: VPC-Prod
  Propagated: VPC-Shared-Services, VPN
  → Prod can reach Shared Services and On-Prem, but NOT Dev
```

*"This is network segmentation. Dev accidentally deploying something that reaches Prod? Blocked at the TGW level. Even if someone misconfigures a Security Group, the TGW route table won't allow the traffic."*

### 🗣️ How Traffic Flows Through TGW

```
Step-by-step: EC2 in VPC-Dev (10.0.1.50) → EC2 in VPC-Prod (10.2.3.100)

1. EC2 sends packet: src=10.0.1.50, dst=10.2.3.100
2. VPC-Dev route table: 10.2.0.0/16 → tgw-xxx  (Transit Gateway)
3. TGW receives packet on VPC-Dev attachment
4. TGW checks dev-rt route table:
   - 10.2.0.0/16 → VPC-Prod attachment? 
   - If route EXISTS → forward to VPC-Prod attachment
   - If route DOESN'T EXIST → packet dropped (isolated!)
5. Packet arrives at VPC-Prod's TGW attachment
6. VPC-Prod's route table delivers to 10.2.3.100
7. Security Group & NACL on the destination EC2 are checked
8. If allowed → delivered
```

### 🗣️ Appliance Mode

*"By default, TGW uses ECMP (Equal-Cost Multi-Path) — it distributes traffic across AZs. But if you're routing traffic through a firewall appliance (Palo Alto, Fortinet), this breaks stateful inspection because requests and responses might go through different appliances."*

*"Appliance Mode forces all traffic for a flow to go through the SAME AZ. Enable it on the VPC attachment where your firewall sits."*

## Part 4: Transit Gateway vs VPC Peering (5 minutes)

### 🗣️ When to Use Which

| Feature | VPC Peering | Transit Gateway |
|---------|-------------|-----------------|
| **Max connections** | 125 per VPC | 5,000 per TGW |
| **Transitive** | ❌ No | ✅ Yes |
| **Cross-region** | ✅ Yes | ✅ Yes (TGW Peering) |
| **Cross-account** | ✅ Yes | ✅ Yes (RAM sharing) |
| **Bandwidth** | No limit (AWS backbone) | 50 Gbps per attachment |
| **Cost** | Data transfer only ($0.01/GB) | $0.05/hr + $0.02/GB |
| **Routing** | Manual routes both sides | Centralized TGW route tables |
| **VPN support** | ❌ No | ✅ Yes |
| **Best for** | 2-5 VPCs, simple connectivity | 5+ VPCs, hub-spoke, hybrid cloud |

*"Rule of thumb: 2-5 VPCs with simple connectivity? VPC Peering. More than 5 VPCs, or VPN/Direct Connect? Transit Gateway."*

## Part 7: Site-to-Site VPN Deep Dive (20 minutes)

### 🗣️ The Architecture

```
YOUR OFFICE                              AWS
┌────────────────┐                    ┌──────────────────┐
│                │   IPSec Tunnel 1   │                  │
│  Your Router   │════════════════════│  VPN Connection  │
│ (Customer GW)  │   IPSec Tunnel 2   │                  │
│                │════════════════════│  VGW or TGW      │
│ 192.168.0.0/16 │    (redundancy)    │  10.0.0.0/16     │
└────────────────┘                    └──────────────────┘
     On-Prem                              VPC
```

### 🗣️ Three Components You Need

**1. Customer Gateway (CGW)**
*"This represents YOUR SIDE of the VPN — your office router or firewall."*

| Field | What It Is |
|-------|-----------|
| **Public IP** | The public IP of your office router |
| **BGP ASN** | If using dynamic routing (BGP), your ASN number |
| **Device type** | Optional — helps AWS generate config (Cisco, Juniper, etc.) |

*"This is a LOGICAL resource in AWS. You're registering your router with AWS so it knows where to point the tunnel."*

**2. Virtual Private Gateway (VGW)**
*"This is AWS's SIDE of the VPN — it attaches to your VPC."*

- Attached to ONE VPC
- Can have multiple VPN connections (up to 10)
- Supports both static routing and BGP (dynamic)
- Automatically deploys in two AZs for redundancy

*"Or you can use Transit Gateway instead of VGW — connects the VPN to TGW which then reaches ALL your VPCs."*

**3. VPN Connection**
*"The actual tunnel configuration. AWS creates TWO IPSec tunnels per connection for redundancy."*

```
YOUR ROUTER ═══ Tunnel 1 ═══ AWS Endpoint 1 (AZ-a)
            ═══ Tunnel 2 ═══ AWS Endpoint 2 (AZ-b)
```

*"If Tunnel 1 dies (AZ-a outage), traffic automatically fails over to Tunnel 2 (AZ-b). Always configure BOTH tunnels on your router!"*

### 🗣️ IPSec — What's Actually Happening

*"IPSec creates an encrypted tunnel between two endpoints. Here's the simplified process:"*

```
Phase 1 (IKE — Internet Key Exchange):
  Your router and AWS negotiate:
  ├── Encryption algorithm (AES-256)
  ├── Hashing algorithm (SHA-256)
  ├── Authentication (pre-shared key)
  └── Diffie-Hellman group (key exchange)
  Result: Secure channel established ✓

Phase 2 (IPSec SA — Security Association):
  Both sides agree on:
  ├── What traffic to encrypt (which CIDRs)
  ├── Encryption for data (AES-256)
  ├── Hashing for integrity (SHA-256)
  └── Lifetime (3600 seconds default)
  Result: Data can now flow encrypted ✓
```

*"You don't need to memorize the phases, but know that IPSec has TWO phases and uses AES-256 encryption."*

### 🗣️ Static vs Dynamic Routing (BGP)

| Feature | Static Routing | Dynamic Routing (BGP) |
|---------|---------------|----------------------|
| **Routes** | Manually configured on both sides | Automatically exchanged via BGP |
| **Failover** | Manual (you update routes) | Automatic (BGP detects failure) |
| **Complexity** | Simple | Requires BGP knowledge |
| **Use case** | Small, simple networks | Enterprise, multiple connections |
| **AWS recommendation** | Dev/test only | Production |

*"BGP stands for Border Gateway Protocol. It's how internet routers talk to each other. With BGP, if you add a new subnet on-prem, your router automatically tells AWS about it. No manual route updates."*

### 🗣️ BGP ASN (Autonomous System Number)

*"Every organization running BGP needs an ASN — a unique number identifying your network."*

| ASN Type | Range | Use |
|----------|-------|-----|
| **Public ASN** | 1 – 64,495 | Assigned by IANA/RIR (for internet routing) |
| **Private ASN** | 64,512 – 65,534 | For internal/VPN use (no IANA registration) |
| **AWS Default** | 64,512 | Default VGW ASN |

*"For VPN labs, use a private ASN like 65,000. AWS VGW defaults to 64,512."*

## Part 8: VPN Tunnels — Advanced Details (10 minutes)

### 🗣️ Tunnel Options

*"Each VPN tunnel has configurable options:"*

| Option | Default | What It Does |
|--------|---------|-------------|
| **Inside tunnel CIDR** | /30 from 169.254.x.x | Link-local IPs for the tunnel endpoints |
| **Pre-shared key** | Auto-generated | The password for IKE Phase 1 |
| **IKE version** | IKEv2 | IKEv1 (older) or IKEv2 (preferred) |
| **Phase 1 encryption** | AES-256 | AES-128, AES-256, AES-256-GCM |
| **Phase 1 integrity** | SHA-256 | SHA-1, SHA-256, SHA-384, SHA-512 |
| **Phase 1 DH group** | 14, 15, 16, 17, 18, 19, 20, 21 | Higher = more secure, more CPU |
| **Phase 1 lifetime** | 28,800 sec (8 hrs) | How often keys renegotiate |
| **Phase 2 encryption** | AES-256 | Same options as Phase 1 |
| **Phase 2 integrity** | SHA-256 | Same options as Phase 1 |
| **Phase 2 lifetime** | 3,600 sec (1 hr) | How often data keys renegotiate |
| **Dead Peer Detection** | Enabled (10 sec) | Detects if the other side is down |
| **Startup action** | Add | `Add` (wait for traffic) or `Start` (immediately initiate) |
| **Logging** | Disabled | Send IKE logs to CloudWatch |

### 🗣️ Inside Tunnel IPs

*"Each tunnel gets a /30 CIDR from the 169.254.x.x range — that gives exactly 4 IPs, 2 usable:"*

```
Tunnel 1: 169.254.10.0/30
  169.254.10.0 — Network (unusable)
  169.254.10.1 — AWS side (VGW/TGW endpoint)
  169.254.10.2 — Your side (customer router)
  169.254.10.3 — Broadcast (unusable)
```

*"These IPs are used for BGP peering inside the tunnel. They don't affect your VPC or on-prem addressing."*

### 🗣️ VPN Throughput & Limitations

| Metric | Value |
|--------|-------|
| **Max throughput per tunnel** | 1.25 Gbps |
| **Max packets per second** | 140,000 |
| **Tunnels per connection** | 2 (for redundancy) |
| **Max VPN connections per VGW** | 10 |
| **ECMP with TGW** | Up to 50 Gbps (using 2 tunnels per connection × multiple connections) |
| **MTU** | 1,399 bytes (due to IPSec overhead) |

*"1.25 Gbps per tunnel. If you need more, attach the VPN to a Transit Gateway and enable ECMP (Equal-Cost Multi-Path). With ECMP, traffic spreads across ALL tunnels from ALL connections. 4 VPN connections = 8 tunnels = up to 10 Gbps aggregate."*

## Part 9: VPN with Transit Gateway vs VGW (10 minutes)

### 🗣️ The Two Options

```
Option A: VPN → Virtual Private Gateway → ONE VPC
  └── Simple, good for single-VPC setups

Option B: VPN → Transit Gateway → MANY VPCs
  └── Enterprise, good for hub-spoke architectures
```

### 🗣️ Comparison

| Feature | VPN + VGW | VPN + TGW |
|---------|-----------|-----------|
| **Reaches** | 1 VPC only | All VPCs attached to TGW |
| **ECMP** | ❌ No (1.25 Gbps max) | ✅ Yes (aggregate up to 50 Gbps) |
| **Routing** | Static or BGP | Static or BGP |
| **Route propagation** | To VPC route table | To TGW route table |
| **Cost** | $0.05/hr VPN only | $0.05/hr VPN + $0.05/hr TGW attachment |
| **Best for** | Small setups, single VPC | Enterprise, multi-VPC |

### 🗣️ VPN + TGW ECMP

*"This is the killer feature. With VGW, you're capped at 1.25 Gbps. With TGW + ECMP:"*

```
VPN Connection 1:
  Tunnel 1 → 1.25 Gbps
  Tunnel 2 → 1.25 Gbps
VPN Connection 2:
  Tunnel 3 → 1.25 Gbps
  Tunnel 4 → 1.25 Gbps
────────────────────────
Aggregate:  5 Gbps with ECMP enabled on TGW
```

*"Just create multiple VPN connections from the same Customer Gateway to the same TGW. TGW automatically load-balances across all tunnels."*

## Part 10: AWS Client VPN (10 minutes)

### 🗣️ What is Client VPN?

*"Site-to-Site VPN connects your entire OFFICE network. Client VPN connects individual LAPTOPS or phones."*

```
Employee laptop (anywhere)
    │
    └── OpenVPN Client → AWS Client VPN Endpoint → VPC
```

### 🗣️ How It Works

| Component | What It Is |
|-----------|-----------|
| **Client VPN Endpoint** | The AWS-managed VPN server (lives in your VPC) |
| **Target Network** | The VPC subnet(s) clients can access |
| **Authorization Rules** | Who can access what (by CIDR, by AD group) |
| **Client CIDR** | IP range assigned to connected clients (e.g., 10.100.0.0/16) |
| **Authentication** | Mutual TLS (certificates), Active Directory, or SAML/SSO |

### 🗣️ Client VPN Authentication Options

| Method | How It Works | Best For |
|--------|-------------|----------|
| **Mutual TLS** | Client + server both have certificates | Small teams, no AD |
| **Active Directory** | Username/password against AD/LDAP | Enterprises with AD |
| **SAML/SSO** | Federated login (Okta, Azure AD) | Modern enterprises |
| **Mutual TLS + AD** | Certificate AND username/password | Maximum security |

### 🗣️ Pricing

```
Client VPN Endpoint:  $0.10/hour (~$73/month) — whether anyone is connected or not
Per connection:       $0.05/hour per active connection

5 developers connected 8 hours/day, 22 days/month:
  Endpoint: $73/month
  Connections: 5 × $0.05 × 8 × 22 = $44/month
  Total: $117/month
```

## Part 11: Direct Connect vs VPN (5 minutes)

### 🗣️ Quick Comparison

| Feature | Site-to-Site VPN | AWS Direct Connect |
|---------|-----------------|-------------------|
| **Connection** | Over public internet (encrypted) | Dedicated physical fiber |

*"Most companies use BOTH: Direct Connect as primary (for performance) and VPN as backup (for failover). If Direct Connect goes down, traffic automatically routes through the VPN."*

## Part 12: Monitoring VPN & TGW (5 minutes)

### 🗣️ CloudWatch Metrics

**VPN Metrics:**
| Metric | What It Shows |
|--------|--------------|
| `TunnelState` | 1 = UP, 0 = DOWN |
| `TunnelDataIn` | Bytes received from on-prem |
| `TunnelDataOut` | Bytes sent to on-prem |

**Transit Gateway Metrics:**
| Metric | What It Shows |
|--------|--------------|
| `BytesIn` / `BytesOut` | Data flowing through each attachment |
| `PacketDropCountBlackhole` | Packets dropped because no route exists |
| `PacketsIn` / `PacketsOut` | Packet counts per attachment |

### 🗣️ Essential Alarms

```
Alarm 1: VPN Tunnel Down
  Metric: TunnelState < 1
  Period: 1 minute
  Action: SNS → PagerDuty / Slack

Alarm 2: Both Tunnels Down (Critical!)
  Metric: TunnelState = 0 for BOTH tunnels
  Period: 1 minute
  Action: SNS → Phone call to on-call

Alarm 3: TGW Blackhole Drops
  Metric: PacketDropCountBlackhole > 0
  Period: 5 minutes
  Action: SNS → email (routing misconfiguration)
```

## Part 13: Interview Questions — TGW & VPN (10 minutes)

### 🗣️ Top 20 Interview Questions

**Transit Gateway:**

1. **What is Transit Gateway?**
   → A regional network hub that connects VPCs, VPNs, and Direct Connect through a single central point.

2. **Is Transit Gateway transitive? What about VPC Peering?**
   → TGW is transitive (A→TGW→B works). VPC Peering is NOT transitive.

3. **How do you isolate Dev from Prod in TGW?**
   → Create separate TGW route tables. Associate Dev VPCs with a dev-rt that only has routes to Shared Services. Don't propagate Prod routes to dev-rt.

4. **What is the max bandwidth per TGW VPC attachment?**
   → 50 Gbps (burst). Sustained is typically 10-20 Gbps.

5. **Can TGW connect VPCs across regions?**
   → Yes, using Transit Gateway Peering (inter-region).

6. **What is TGW Appliance Mode?**
   → Forces request/response for the same flow to go through the same AZ. Required for stateful firewall appliances.

7. **How does TGW route table differ from VPC route table?**
   → TGW route tables use Associations (which attachment uses this table) and Propagations (which attachments advertise routes). VPC route tables just have destination→target.

8. **TGW pricing?**
   → $0.05/hr per attachment + $0.02/GB processed. A 5-VPC setup costs ~$185/month.

**VPN:**

9. **How many tunnels does a VPN connection have?**
   → 2 tunnels (for redundancy across AZs). Always configure both.

10. **What protocol does Site-to-Site VPN use?**
    → IPSec (IKEv1 or IKEv2).

11. **Max throughput per VPN tunnel?**
    → 1.25 Gbps. Use ECMP with TGW for more.

12. **What is a Customer Gateway?**
    → A logical AWS resource representing your on-prem router. Contains the public IP and optional BGP ASN.

13. **Static vs BGP routing for VPN?**
    → Static: manual route configuration. BGP: routes exchange automatically. AWS recommends BGP for production (automatic failover).

14. **What is ECMP?**
    → Equal-Cost Multi-Path. TGW distributes traffic across all available VPN tunnels for higher aggregate throughput.

15. **Can VPN with VGW reach multiple VPCs?**
    → No. VGW attaches to ONE VPC only. Use TGW if you need multiple VPCs.

16. **Difference between Site-to-Site VPN and Client VPN?**
    → Site-to-Site: network-to-network (office router ↔ AWS). Client VPN: individual device ↔ AWS (uses OpenVPN).

17. **Is Direct Connect encrypted?**
    → No! It's a private connection but NOT encrypted. To encrypt Direct Connect traffic, run a VPN on top of it.

18. **What happens if one VPN tunnel goes down?**
    → Traffic automatically fails over to the second tunnel. This is why you must configure both tunnels.

19. **What is Dead Peer Detection (DPD)?**
    → A mechanism that detects if the remote VPN endpoint is unreachable. Default: 10-second interval, 3 retries. After 30 seconds, the tunnel is marked DOWN.

20. **How do you monitor VPN tunnel state?**
    → CloudWatch metric `TunnelState` (1=UP, 0=DOWN). Set alarms for `TunnelState < 1`.

# SECTION C: HANDS-ON LABS

## 🟢 Lab 1: BASIC — Transit Gateway Connecting 2 VPCs (30 minutes)

*"This is the simplest TGW lab. Two VPCs that can't communicate get connected through Transit Gateway."*

### Architecture

```
┌─────────────┐                     ┌─────────────┐
│  VPC-Dev     │                     │  VPC-Prod   │
│ 10.0.0.0/16  │──── Transit ────────│ 10.1.0.0/16 │
│              │     Gateway         │             │
│ EC2: 10.0.1.x│                     │ EC2: 10.1.1.x│
└─────────────┘                     └─────────────┘
```

### Step 1: Create VPC-Dev

1. **VPC** → **Create VPC** → VPC only
   - Name: `vpc-dev`
   - CIDR: `10.0.0.0/16`
2. **Create subnet:**
   - Name: `dev-private-1a`
   - AZ: `us-east-1a`
   - CIDR: `10.0.1.0/24`

### Step 2: Create VPC-Prod

1. **VPC** → **Create VPC** → VPC only
   - Name: `vpc-prod`
   - CIDR: `10.1.0.0/16`
2. **Create subnet:**
   - Name: `prod-private-1a`
   - AZ: `us-east-1a`
   - CIDR: `10.1.1.0/24`

### Step 3: Launch EC2 Instances

In each VPC, launch a t2.micro EC2:
- **VPC-Dev:** Name `dev-server`, subnet `dev-private-1a`, SG allows ICMP (ping) from `10.0.0.0/8`
- **VPC-Prod:** Name `prod-server`, subnet `prod-private-1a`, SG allows ICMP from `10.0.0.0/8`

> Use **SSM Session Manager** to access these instances (no public IP needed). Ensure the instance profile has `AmazonSSMManagedInstanceCore` policy.

### Step 4: Verify — They CANNOT Talk

```bash
# From dev-server (10.0.1.x), try to ping prod-server:
ping 10.1.1.x -c 3
# → 100% packet loss! They're in different VPCs.
```

### Step 5: Create Transit Gateway

1. **VPC** → **Transit Gateways** → **Create**
   - Name: `demo-tgw`
   - ASN: `64512` (default)
   - DNS support: ✅ Enable
   - Default route table association: ✅ Enable
   - Default route table propagation: ✅ Enable
2. Wait 2-3 minutes for TGW to become `Available`

### Step 6: Attach Both VPCs

1. **Transit Gateway Attachments** → **Create**
   - Name: `dev-attachment`
   - Transit Gateway: `demo-tgw`
   - Attachment type: VPC
   - VPC: `vpc-dev`
   - Subnet: `dev-private-1a`

2. Create another:
   - Name: `prod-attachment`
   - Transit Gateway: `demo-tgw`
   - VPC: `vpc-prod`
   - Subnet: `prod-private-1a`

3. Wait 2-3 minutes for both to become `Available`

### Step 7: Update VPC Route Tables

**VPC-Dev route table:**
1. Select the route table for `dev-private-1a`
2. **Edit routes** → **Add route:**
   - Destination: `10.1.0.0/16`
   - Target: Transit Gateway → `demo-tgw`

**VPC-Prod route table:**
1. Select the route table for `prod-private-1a`
2. **Edit routes** → **Add route:**
   - Destination: `10.0.0.0/16`
   - Target: Transit Gateway → `demo-tgw`

### Step 8: Verify — They CAN Talk Now!

```bash
# From dev-server:
ping 10.1.1.x -c 3
# → SUCCESS! 0% packet loss!

# From prod-server:
ping 10.0.1.x -c 3
# → SUCCESS!
```

### What Students Learned:

*"You just connected two completely isolated VPCs through Transit Gateway. Without TGW, these VPCs had zero ability to communicate. Now they can talk via private IPs."*

## 🟡 Lab 2: INTERMEDIATE — TGW with Network Segmentation (30 minutes)

*"This lab adds a third VPC (Shared Services) and demonstrates how to ISOLATE Dev from Prod while both can reach Shared Services."*

### Architecture

```
                    ┌──────────────┐
                    │ VPC-Shared   │
                    │ 10.2.0.0/16  │
                    │ (Monitoring) │
                    └──────┬───────┘
                           │
┌─────────────┐    ┌───────┴──────┐    ┌─────────────┐
│  VPC-Dev     │    │  Transit     │    │  VPC-Prod   │
│ 10.0.0.0/16  │────│  Gateway     │────│ 10.1.0.0/16 │
│              │    │              │    │             │
└─────────────┘    └──────────────┘    └─────────────┘

✅ Dev → Shared:    ALLOWED
✅ Prod → Shared:   ALLOWED
❌ Dev → Prod:      BLOCKED
❌ Prod → Dev:      BLOCKED
```

### Step 1: Add VPC-Shared (if not already created)

1. Create VPC: `vpc-shared`, CIDR: `10.2.0.0/16`
2. Create subnet: `shared-private-1a`, CIDR: `10.2.1.0/24`
3. Launch EC2: `shared-server`
4. Create TGW attachment: `shared-attachment`

### Step 2: Create Custom TGW Route Tables

**Delete or stop using the default association/propagation** — we'll create our own.

1. **Transit Gateway Route Tables** → **Create:**
   - Name: `dev-rt`
   - TGW: `demo-tgw`

2. Create another:
   - Name: `prod-rt`

3. Create another:
   - Name: `shared-rt`

### Step 3: Set Up Associations

| TGW Route Table | Associated Attachment |
|----------------|-----------------------|
| `dev-rt` | `dev-attachment` |
| `prod-rt` | `prod-attachment` |
| `shared-rt` | `shared-attachment` |

For each: Select TGW Route Table → **Associations** tab → **Create association** → Select attachment

### Step 4: Set Up Propagations

| TGW Route Table | Propagated From |
|----------------|-----------------|
| `dev-rt` | `shared-attachment` only |
| `prod-rt` | `shared-attachment` only |
| `shared-rt` | `dev-attachment` AND `prod-attachment` |

For each: Select TGW Route Table → **Propagations** tab → **Create propagation** → Select attachment

### Step 5: Update VPC Route Tables

**VPC-Dev route table:** Add `10.2.0.0/16 → tgw`
**VPC-Prod route table:** Add `10.2.0.0/16 → tgw`
**VPC-Shared route table:** Add `10.0.0.0/16 → tgw` AND `10.1.0.0/16 → tgw`

### Step 6: Verify Segmentation

```bash
# From dev-server:
ping 10.2.1.x -c 3    # → Shared: SUCCESS ✅
ping 10.1.1.x -c 3    # → Prod:   FAIL ❌ (packet loss!)

# From prod-server:
ping 10.2.1.x -c 3    # → Shared: SUCCESS ✅
ping 10.0.1.x -c 3    # → Dev:    FAIL ❌ (packet loss!)

# From shared-server:
ping 10.0.1.x -c 3    # → Dev:    SUCCESS ✅
ping 10.1.1.x -c 3    # → Prod:   SUCCESS ✅
```

### What Students Learned:

*"Dev can reach Shared Services (for logging, monitoring). Prod can reach Shared Services. But Dev CANNOT reach Prod — even through TGW. This is network segmentation."*

*"Even if a developer in Dev gets compromised, they can't pivot to Production. This is how banks and healthcare companies isolate environments."*

## 🔴 Lab 3: ADVANCED — TGW + VPN + Simulated On-Prem (45 minutes)

*"This is the real-world scenario. We'll simulate an on-premises data center using a separate VPC with a software VPN (strongSwan), connect it to TGW via Site-to-Site VPN, and verify end-to-end connectivity."*

### Architecture

```
"On-Prem" (Simulated)                    AWS Cloud
┌──────────────────┐                   ┌──────────────┐
│ VPC-OnPrem       │    IPSec VPN      │  Transit     │
│ 192.168.0.0/16   │═══════════════════│  Gateway     │
│                  │   2 tunnels       │              │
│ EC2 + strongSwan │                   │  ┌────────┐  │
│ (Customer GW)    │                   │  │VPC-Dev │  │
└──────────────────┘                   │  │10.0/16 │  │
                                       │  ├────────┤  │
                                       │  │VPC-Prod│  │
                                       │  │10.1/16 │  │
                                       │  └────────┘  │
                                       └──────────────┘
```

### Step 1: Create "On-Prem" VPC

1. Create VPC: `vpc-onprem`, CIDR: `192.168.0.0/16`
2. Create public subnet: `onprem-public`, CIDR: `192.168.1.0/24`
3. Create Internet Gateway → Attach to `vpc-onprem`
4. Route table: `0.0.0.0/0 → igw` (needs internet for VPN tunnel)

### Step 2: Launch strongSwan EC2

1. Launch EC2 in `onprem-public`:
   - Name: `onprem-router`
   - AMI: Amazon Linux 2023
   - Type: `t3.micro`
   - Public IP: ✅ Enable
   - SG inbound: SSH(22) from your IP, ALL from `192.168.0.0/16`, Custom UDP(500, 4500) from `0.0.0.0/0`

2. **CRITICAL:** Disable Source/Dest Check:
   - EC2 → Select instance → Actions → Networking → **Change source/destination check** → **Stop** → Save

   *"This is required because the instance acts as a router — forwarding packets not addressed to itself."*

3. SSH in and install strongSwan:
```bash
sudo yum install -y strongswan
sudo systemctl enable strongswan
```

### Step 3: Create Customer Gateway in AWS

1. **VPC** → **Customer Gateways** → **Create**
   - Name: `onprem-cgw`
   - Routing: `Dynamic` (BGP)
   - BGP ASN: `65000`
   - IP Address: Public IP of `onprem-router` EC2

### Step 4: Create VPN Connection

1. **VPC** → **Site-to-Site VPN Connections** → **Create**
   - Name: `onprem-vpn`
   - Target Gateway: Select `demo-tgw` (Transit Gateway)
   - Customer Gateway: `onprem-cgw`
   - Routing: Dynamic (BGP)
   - Inside tunnel 1 CIDR: `169.254.10.0/30`
   - Inside tunnel 2 CIDR: `169.254.10.4/30`
   - Local IPv4: `192.168.0.0/16`
   - Remote IPv4: `10.0.0.0/8`

2. Wait for VPN to be created

3. **Download Configuration:**
   - Select VPN → **Download Configuration**
   - Vendor: `Generic` or `strongSwan`
   - This gives you the pre-shared keys and tunnel IPs

### Step 5: Configure strongSwan

SSH into `onprem-router` and create the config:

```bash
# Get these values from the downloaded VPN configuration
TUNNEL1_AWS_IP="<AWS Tunnel 1 Outside IP>"
TUNNEL1_PSK="<Pre-Shared Key from download>"
TUNNEL1_AWS_INSIDE="169.254.10.1"
TUNNEL1_LOCAL_INSIDE="169.254.10.2"

sudo tee /etc/strongswan/ipsec.conf > /dev/null <<EOF
config setup
    charondebug="ike 2, knl 2, cfg 2"

conn tunnel1
    auto=start
    type=tunnel
    authby=secret
    left=%defaultroute
    leftid=$(curl -s http://169.254.169.254/latest/meta-data/public-ipv4)
    leftsubnet=192.168.0.0/16
    right=${TUNNEL1_AWS_IP}
    rightsubnet=10.0.0.0/8
    ike=aes256-sha256-modp2048
    esp=aes256-sha256-modp2048
    keyexchange=ikev2
    ikelifetime=8h
    lifetime=1h
    dpddelay=10s
    dpdtimeout=30s
    dpdaction=restart
EOF

sudo tee /etc/strongswan/ipsec.secrets > /dev/null <<EOF
$(curl -s http://169.254.169.254/latest/meta-data/public-ipv4) ${TUNNEL1_AWS_IP} : PSK "${TUNNEL1_PSK}"
EOF

# Enable IP forwarding
echo "net.ipv4.ip_forward = 1" | sudo tee -a /etc/sysctl.conf
sudo sysctl -p

# Start strongSwan
sudo systemctl restart strongswan
```

### Step 6: Verify VPN Tunnel is UP

```bash
# On the onprem-router EC2:
sudo strongswan status
# → Should show: tunnel1[1]: ESTABLISHED

# In AWS Console:
# VPC → Site-to-Site VPN → Select your VPN → Tunnel Details
# Tunnel 1 Status: UP ✅
```

### Step 7: Update TGW Route Table

The VPN attachment is auto-propagated if you enabled default propagation. Otherwise:

1. **TGW Route Tables** → Select the appropriate table
2. **Propagations** → Create → Select VPN attachment
3. Verify route `192.168.0.0/16 → vpn-attachment` appears

### Step 8: Update VPC Route Tables

In VPC-Dev and VPC-Prod route tables, add:
- Destination: `192.168.0.0/16`
- Target: Transit Gateway

In the onprem VPC route table, add:
- Destination: `10.0.0.0/8`
- Target: Instance → `onprem-router` (since it's the VPN endpoint)

### Step 9: Test End-to-End

```bash
# From dev-server (10.0.1.x):
ping 192.168.1.x -c 3
# → SUCCESS! Dev VPC reaching "on-prem" through TGW → VPN tunnel!

# From onprem EC2 (192.168.1.x):
ping 10.0.1.x -c 3
# → SUCCESS! On-prem reaching AWS through VPN → TGW → VPC!

# From prod-server (10.1.1.x):
ping 192.168.1.x -c 3
# → SUCCESS! (if prod-rt has VPN propagation)
```

### Step 10: Show Students the Flow

*"Let's trace the exact path of a packet from on-prem to Dev:"*

```
1. On-prem server (192.168.1.x) sends packet to 10.0.1.x
2. On-prem route table: 10.0.0.0/8 → onprem-router
3. onprem-router encrypts packet with IPSec
4. Encrypted packet travels through the INTERNET to AWS
5. AWS decrypts at the VPN endpoint
6. Packet enters TGW on the VPN attachment
7. TGW route table: 10.0.0.0/16 → dev-attachment
8. Packet enters VPC-Dev
9. VPC-Dev route table delivers to 10.0.1.x
10. Security Group allows ICMP → delivered! ✅
```
